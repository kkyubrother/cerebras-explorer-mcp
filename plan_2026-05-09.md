확인했습니다. 이번 수정본은 **이전 P0 지적사항의 핵심을 대부분 제대로 반영했습니다.**
제가 의도했던 방향, 즉 **“runtime raw result는 풍부하게 유지하되, MCP agent-facing `structuredContent`는 compact contract로 분리한다”**는 핵심 구조가 이제 실제 코드에 들어갔습니다. 이전 검토 기준과 비교해 보면 가장 큰 blocker들은 해결된 상태입니다.

다만 아직 몇 가지 문서/벤치마크/입력 schema 정리 이슈가 남아 있습니다. 기능 핵심보다는 **정합성 마무리**에 가깝습니다.

## 3.6 `hasEditIntent()`가 약간 넓습니다

현재 edit intent detection은 다음 regex입니다.

근거: `src/explorer/runtime.mjs:348-351`.

```js
/fix|change|update|modify|implement|add|remove|refactor|migrate|patch|수정|변경|구현|추가|삭제|리팩터|마이그레이션/
```

이 자체는 이해됩니다. 하지만 `change`가 넓어서 다음처럼 edit planning이 아닌 질문도 `targeted_read_needed`로 갈 수 있습니다.

* “what changed recently?”
* “review change context”
* “summarize changes”
* “explain how behavior changes across versions”

특히 `review_change_context`의 task builder는 `"Review change context..."`로 시작합니다. 근거: `src/mcp/server.mjs:526-543`. 이 경우 edit 의도가 없어도 `hasEditIntent()`가 true가 될 수 있습니다.

현재는 legacy `summarize_changes`가 기본 비노출이므로 큰 문제는 줄었지만, `review_change_context`에서는 `verified` 또는 `follow_up_needed`가 더 적절한 경우도 있을 수 있습니다.

권장 개선:

```js
function hasEditIntent(task) {
  const text = String(task ?? '').toLowerCase();

  if (/review change context|what changed|summarize changes|recent changes/.test(text)) {
    return false;
  }

  return /\b(fix|modify|implement|refactor|migrate|patch)\b/.test(text)
    || /\b(add|remove|update|change)\b.*\b(code|field|schema|behavior|implementation|tool|api)\b/.test(text)
    || /수정|변경|구현|추가|삭제|리팩터|마이그레이션/.test(text);
}
```

더 좋은 방법은 wrapper builder가 task mode를 명시하는 것입니다.

```js
hints: {
  strategy: 'reference-chase',
  taskMode: 'edit_planning'
}
```

또는 internal-only metadata로 `exploreArgs.taskMode = 'edit_planning' | 'review' | 'evidence' | 'explanation'`를 넘겨 `buildResultStatus()`가 regex 대신 task mode를 쓰게 만드는 방식이 더 안정적입니다.

# 최종 평가

이번 수정은 이전보다 훨씬 좋아졌습니다. 특히 아래 네 가지가 들어간 점이 결정적입니다.

1. `structuredContent` compact 변환
2. top-level `sessionId`
3. legacy tools opt-in
4. targets merge + evidence-derived `read` role

이제 Codex/Claude Code 입장에서 결과를 읽는 흐름이 꽤 명확해졌습니다.

```txt
directAnswer → status → targets → evidence → nextAction → sessionId
```

남은 문제는 대부분 “구현 방향이 틀렸다”가 아니라, **문서와 benchmark가 새 compact contract를 완전히 따라오지 못한 것**입니다.
따라서 다음 수정은 큰 구조 변경보다, integration/README/DESIGN/evaluator를 새 contract에 맞춰 정렬하는 작업이 가장 효과적입니다.
