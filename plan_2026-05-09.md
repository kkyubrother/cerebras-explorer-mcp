확인했습니다. 결론부터 말하면 **제가 의도한 방향의 핵심은 많이 반영됐지만, “agent-facing 입력/출력을 정말 단순하게 만든다”는 관점에서는 아직 절반 정도 남아 있습니다.**

좋게 반영된 부분은 분명합니다. `budget/thoroughness`를 advanced/legacy로 낮췄고, 목적형 wrapper를 추가했고, `explore_v2`를 기본 비노출로 바꿨고, `directAnswer/status/targets/evidence snippet/nextAction/_debug` 같은 v3 성격의 필드가 실제 runtime과 MCP schema에 들어갔습니다. 테스트도 `npm test` 기준 **259개 중 258 pass, 1 skip, fail 0**으로 통과했습니다.

다만 현재 구조는 아직 상위 agent에게 “간결한 계약”만 주는 것이 아니라, **v3 필드 위에 legacy/debug 필드가 그대로 top-level에 남아 있는 상태**입니다. 이 때문에 Codex/Claude Code가 다시 `stats`, `candidatePaths`, `followups`, `confidenceScore`, `codeMap` 등을 읽고 판단하려 할 가능성이 남아 있습니다.

# 3. 아직 의도와 어긋나는 부분

## 3.1 가장 큰 문제: top-level output이 아직 너무 무겁습니다

제가 제안한 핵심은 “agent-facing top-level에는 꼭 필요한 것만 남기고, 나머지는 `_debug` 또는 legacy로 내리자”였습니다.

그런데 현재 runtime은 `_debug`를 추가하면서도 기존 top-level 필드를 제거하지 않습니다.

예를 들어 최종화에서 다음 필드들이 여전히 top-level에 남습니다.

* `answer`
* `summary`
* `confidence`
* `candidatePaths`
* `followups`
* `stats`
* `confidenceScore`
* `confidenceLevel`
* `confidenceFactors`
* `critic`
* `codeMap`
* `diagram`
* `recentActivity`
* `trustSummary`

특히 `codeMap`, `diagram`, `recentActivity`는 `_debug`에 들어가지만 동시에 top-level에도 남습니다.

근거:

* `stats`는 `normalizeExploreResult()`에서 top-level로 반환됨: `src/explorer/schemas.mjs:440`
* `codeMap` top-level 주입: `src/explorer/runtime.mjs:1363-1372`
* `recentActivity` top-level 주입: `src/explorer/runtime.mjs:1375-1377`
* `_debug`는 추가만 하고 기존 top-level을 제거하지 않음: `src/explorer/runtime.mjs:466-478`, `src/explorer/runtime.mjs:1378`

`examples/expected-response.json`도 이 문제를 보여줍니다. v3 필드가 위에 있긴 하지만, 아래에 `confidenceScore`, `confidenceFactors`, `candidatePaths`, `critic`, `_debug`, `stats`가 모두 top-level에 있습니다. 근거: `examples/expected-response.json:46-127`.

### 왜 문제가 되는가

Codex/Claude Code는 structuredContent를 읽을 때 “어떤 필드가 중요한가”를 추론합니다. top-level에 필드가 많으면 모델은 다음처럼 행동하기 쉽습니다.

* `targets`가 있는데도 `candidatePaths`를 다시 해석함
* `status`가 있는데도 `confidenceScore`를 읽고 자체 판단함
* `_debug.stats`가 있는데도 top-level `stats`를 읽음
* `nextAction`이 있는데도 `followups`를 읽음
* `codeMap`/`diagram`을 실제 dependency graph처럼 과신하거나 다시 검증함

즉, v3 필드가 추가됐지만 **legacy/debug noise가 아직 adoption 이득을 깎고 있습니다.**

### 권장 수정

MCP `structuredContent`는 runtime raw result를 그대로 주지 말고, server layer에서 compact result로 변환하는 것이 좋습니다.

```js
function toAgentFacingResult(result) {
  const sessionId = result.sessionId ?? result.stats?.sessionId ?? null;

  return {
    directAnswer: result.directAnswer || result.answer || '',
    status: result.status ?? {
      confidence: result.confidence ?? 'low',
      verification: 'broad_search_needed',
      complete: false,
      warnings: [],
    },
    targets: result.targets ?? [],
    evidence: result.evidence ?? [],
    uncertainties: result.uncertainties ?? [],
    nextAction: result.nextAction ?? { type: 'stop', reason: '' },
    ...(sessionId ? { sessionId } : {}),
    _debug: {
      ...(result._debug ?? {}),
      legacy: {
        answer: result.answer,
        summary: result.summary,
        candidatePaths: result.candidatePaths,
        followups: result.followups,
        critic: result.critic,
        trustSummary: result.trustSummary,
      },
    },
  };
}
```

그리고 `src/mcp/server.mjs:647-650`를 다음처럼 바꾸는 방향을 권합니다.

```js
const agentResult = toAgentFacingResult(result);

return {
  content: [{ type: 'text', text: formatExploreResult(agentResult) }],
  structuredContent: agentResult,
};
```

runtime direct API 사용자는 기존 full result를 계속 받을 수 있고, MCP agent-facing surface만 가벼워집니다. 이게 가장 backward-compatible합니다.

## 3.2 `sessionId`가 아직 top-level이 아닙니다

현재는 계속 `stats.sessionId`를 쓰도록 되어 있습니다.

* tool description: `src/mcp/server.mjs:27`
* server instructions: `src/mcp/server.mjs:717`
* formatter: `src/mcp/server.mjs:628-630`
* README: `README.md:156`
* Codex AGENTS: `integrations/codex/AGENTS.md.example:11`
* Claude agent: `integrations/claude/.claude/agents/cerebras-explorer.md:23`

### 왜 문제가 되는가

제가 의도한 방향은 `stats`를 debug로 내리는 것이었습니다. 그런데 후속 호출에 필요한 session ID를 `stats.sessionId`에 계속 두면, parent agent가 `stats`를 읽어야 합니다. 그러면 `stats.budget`, `stats.turns`, `stats.toolCalls`, `stats.remainingCalls` 같은 debug 값도 같이 읽게 됩니다.

### 권장 수정

top-level에 `sessionId`를 추가하고, 문구를 모두 바꾸는 게 좋습니다.

```json

```

문구:

```md
Reuse `sessionId` as `session` for follow-up calls.
Do not inspect `_debug.stats` unless debugging explorer behavior.
```

기존 `stats.sessionId`는 legacy로 유지해도 됩니다.

## 3.3 `targets` 생성 로직이 너무 거칠고, model-provided targets를 덮어씁니다

현재 `normalizeExploreResult()`는 model이 낸 `targets`를 정규화합니다. 근거: `src/explorer/schemas.mjs:374-376`.

그런데 최종 runtime에서 다시:

```js
normalized.targets = buildTargets({
  evidence: normalized.evidence,
  candidatePaths: normalized.candidatePaths,
  task: args.task,
});
```

로 덮어씁니다. 근거: `src/explorer/runtime.mjs:1351-1355`.

즉, model이 `targets`에 semantic role을 잘 넣어도 runtime이 버립니다.

또한 `buildTargets()`는 task에 edit intent가 있으면 모든 evidence path를 `edit`으로 만듭니다.

```js
role: editIntent ? 'edit' : 'read'
```

근거: `src/explorer/runtime.mjs:371-380`.

### 왜 문제가 되는가

`map_change_impact` 같은 task에서는 evidence가 다음을 모두 포함할 수 있습니다.

* 실제 수정 후보
* caller
* test
* config consumer
* reference-only file

그런데 task에 “change”, “modify”, “refactor” 같은 단어가 있으면 모든 evidence file이 `role: edit`이 될 수 있습니다. 그러면 parent agent가 너무 많은 파일을 수정 후보로 오해할 수 있습니다.

### 권장 수정

`targets`는 model-provided target을 우선하고, runtime-derived target은 보강용으로 merge해야 합니다.

```js
const modelTargets = normalized.targets ?? [];
const derivedTargets = buildTargets({
  evidence: normalized.evidence,
  candidatePaths: normalized.candidatePaths,
  task: args.task,
});

normalized.targets = mergeTargets(modelTargets, derivedTargets);
```

그리고 `buildTargets()`의 기본 role은 `read`가 맞습니다. `edit`은 다음 조건일 때만 붙이는 편이 안전합니다.

* model이 명시적으로 `role: "edit"`을 낸 경우
* wrapper가 `find_relevant_code`가 아니라 `map_change_impact`이고, target reason에 “likely edit”, “modify here”, “implementation to change” 같은 signal이 있는 경우
* known target file로 입력된 경우

추가로 role enum을 조금 늘리는 것도 권장합니다.

현재 schema는 `edit|read|reference`뿐입니다. 근거: `src/explorer/schemas.mjs:111`.

권장:

```json
"role": "edit|read|test|config|context|reference"
```

이렇게 해야 agent가 “수정 파일”과 “테스트 파일”을 혼동하지 않습니다.

## 3.4 `status.verification`이 너무 쉽게 `targeted_read_needed`가 됩니다

현재 `buildResultStatus()`는 evidence가 있고 critic pass이며 target이 있으면 `targeted_read_needed`로 갑니다.

근거: `src/explorer/runtime.mjs:423-431`.

문제는 `buildTargets()`가 evidence마다 `read` target을 만들기 때문에, evidence가 있는 대부분의 정상 결과가 `targeted_read_needed`가 됩니다.

그러면 `buildNextAction()`이 다시 `read_target`을 권합니다.

근거: `src/explorer/runtime.mjs:441-449`.

### 왜 문제가 되는가

`collect_evidence`나 설명형 질문에서는 exact evidence snippet이 있으면 parent가 broad search는 물론이고 추가 read도 안 해도 됩니다. 그런데 현재 status가 `targeted_read_needed`가 되면 parent가 “그래도 읽어야 하는구나”라고 해석할 수 있습니다.

이건 중복 read 감소 목표와 충돌합니다.

### 권장 수정

`verification` 정책을 task 목적별로 나누는 것이 좋습니다.

```js
function buildResultStatus(result, stats, { taskMode }) {
  if (!hasEvidence || criticFail || stats.stoppedByErrors) {
    return broad_search_needed;
  }

  if (criticCaution || stats.stoppedByBudget) {
    return follow_up_needed;
  }

  if (taskMode === 'edit_planning' || result.targets.some(t => t.role === 'edit')) {
    return targeted_read_needed;
  }

  return verified;
}
```

즉:

| 상황                                                                      | verification           |
| ----------------------------------------------------------------------- | ---------------------- |
| pure explanation / collect evidence / symbol answer with exact snippets | `verified`             |
| edit 전 실제 수정 후보가 있음                                                     | `targeted_read_needed` |
| evidence 부족, budget stop, critic caution                                | `follow_up_needed`     |
| evidence 없음, critic fail, tool error                                    | `broad_search_needed`  |

이렇게 해야 agent가 “검증이 필요한 경우와 필요 없는 경우”를 구분합니다.

## 3.5 legacy tools가 아직 기본 노출되어 tool-selection 비용을 키웁니다

현재 기본 tool list에는 wrapper뿐 아니라 legacy tools도 같이 노출됩니다.

`buildToolList()`에서 wrappers 뒤에 다음 legacy tools가 함께 push됩니다.

* `explain_symbol`
* `trace_dependency`
* `summarize_changes`
* `find_similar_code`

근거: `src/mcp/server.mjs:347-361`.

기본 tools/list 결과도 12개입니다.

```txt
explore_repo
find_relevant_code
trace_symbol
map_change_impact
explain_code_path
collect_evidence
review_change_context
explain_symbol
trace_dependency
summarize_changes
find_similar_code
```

### 왜 문제가 되는가

제 의도는 “목적형 wrapper로 tool choice를 쉽게 만들자”였습니다. 그런데 wrapper와 legacy alias가 동시에 기본 노출되면 agent는 다시 고민합니다.

예:

* `trace_symbol` vs `explain_symbol`
* `map_change_impact` vs `trace_dependency`
* `review_change_context` vs `summarize_changes`
* `find_relevant_code` vs `explore_repo`
* `explore` vs `explore_repo`

12개는 아직 많습니다.

### 권장 수정

기본 노출은 7~8개 이하가 좋습니다.

```txt
find_relevant_code
trace_symbol
map_change_impact
explain_code_path
collect_evidence
review_change_context
explore_repo
```

```js
function legacyToolsEnabled() {

}
```

그리고:

```js
if (legacyToolsEnabled()) {
  tools.push(EXPLAIN_SYMBOL_TOOL, TRACE_DEPENDENCY_TOOL, SUMMARIZE_CHANGES_TOOL, FIND_SIMILAR_CODE_TOOL);
}
```

`find_similar_code`는 유용하므로 wrapper로 계속 둘지, legacy로 둘지 선택하면 됩니다. agent selection을 줄이는 게 목표라면 일단 legacy opt-in이 낫습니다.

## 3.6 일부 integration 문서에 예전 budget policy가 남아 있습니다

좋게 수정된 파일도 있습니다.

* `integrations/codex/AGENTS.md.example:9`는 budget/thoroughness/strategy/language를 설정하지 말라고 합니다.
* `integrations/codex/.agents/skills/cerebras-explore/SKILL.md:25-31`도 좋습니다.
* `integrations/claude/.claude/agents/cerebras-explorer.md:20-24`도 좋습니다.

하지만 hidden integration 파일에 예전 지침이 남아 있습니다.

`integrations/codex/.codex/agents/cerebras_explorer.toml:14-23`에는 아직:

* `budget`, `thoroughness`, `hints`를 justified when task라고 함
* `deep` for initial broad pass
* `normal` for scoped exploration
* `quick` for narrow lookup
* specialized tools behave like internal normal pass, budget choice matters

같은 정책이 남아 있습니다.

`integrations/claude/.claude/skills/cerebras-explore/SKILL.md:19-24`에도 아직:

* `explore_repo`는 first broad pass에 `deep`
* follow-up은 `normal`
* narrow lookup은 `quick`
* `explore`는 `prompt` plus `thoroughness`

가 남아 있습니다.

### 왜 문제가 되는가

이런 skill/agent instruction은 실제 Codex/Claude tool-selection에 매우 강하게 작용합니다. server schema가 “budget 생략”이라고 해도, skill이 “deep/normal/quick 선택”을 말하면 agent는 다시 budget decision을 하게 됩니다.

### 권장 수정

해당 파일들은 아래처럼 바꾸면 됩니다.

```md
Do not choose `budget` or `thoroughness`.
Call the narrowest wrapper with the user query and known anchors only.
The server chooses exploration depth automatically.
Use `sessionId` as `session` for follow-up calls.
```

## 3.7 prompt와 strict schema가 아직 legacy 중심입니다

`src/explorer/prompt.mjs`의 final output contract는 v3 필드를 언급하지만, 여전히 legacy fields를 중심으로 합니다.

근거:

* system prompt output contract: `src/explorer/prompt.mjs:127-141`
* finalize prompt required fields: `src/explorer/prompt.mjs:322-328`

현재 finalize prompt는 다음을 required로 요구합니다.

```txt
answer, summary, confidence, evidence[], candidatePaths[], followups[]
```

근거: `src/explorer/prompt.mjs:322-323`.

또한 `followups`에 `suggestedCall?`을 여전히 허용합니다. 근거: `src/explorer/prompt.mjs:326-327`.

`FOLLOWUP_ITEM_SCHEMA`도 `suggestedCall` 안에 `budget`과 `hints.strategy`를 아직 허용합니다. 근거: `src/explorer/schemas.mjs:67-81`.

### 왜 문제가 되는가

agent-facing에서는 budget을 없애고 있지만, internal final schema가 다시 `suggestedCall.budget`, `suggestedCall.hints.strategy`를 만들 수 있게 합니다. runtime normalization도 suggestedCall에 budget default를 `normal`로 넣습니다. 근거: `src/explorer/schemas.mjs:303-312`.

이건 “복잡도를 output에서 없애자”는 목표와 어긋납니다.

### 권장 수정

모델에게 요구하는 final schema도 더 단순하게 만드는 게 좋습니다.

권장 required:

```txt
answer
confidence
evidence[]
candidatePaths[]
```

optional:

```txt
summary
followups[]
```

그리고 `suggestedCall`은 제거하거나 `query` string만 허용하세요.

```json
{
  "description": "string",
  "priority": "recommended|optional",
  "query": "string"
}
```

`budget`, `hints`, `strategy`, `scope`를 followup output에서 없애야 합니다.

## 3.8 `outputSchema`가 아직 약합니다

`EXPLORE_REPO_OUTPUT_SCHEMA`가 추가된 것은 좋습니다. 근거: `src/explorer/schemas.mjs:156-173`.

이건 compatibility에는 좋지만, MCP client나 agent에게 “상단 핵심 contract”를 강하게 알려주지는 못합니다.

### 권장 수정

MCP용 compact structuredContent를 따로 만들면 output schema도 강하게 만들 수 있습니다.

```js
export const AGENT_FACING_OUTPUT_SCHEMA = {

  additionalProperties: false,

    'directAnswer',
    'status',
```

## 3.9 wrapper input에 아직 불필요한 필드가 남아 있습니다

목적형 wrapper들은 좋아졌지만, 아직 대부분 다음 필드를 갖습니다.

* `language`
* `context`
* `scope`
* `repo_root`
* `session`

예: `find_relevant_code`는 `src/mcp/server.mjs:145-153`에서 `language`와 `context`를 포함합니다. `map_change_impact`, `explain_code_path`, `collect_evidence`, `review_change_context`도 비슷합니다.

### 판단

| 필드          | 판단                                        |
| ----------- | ----------------------------------------- |
| `repo_root` | 유지. multi-root/CLI 환경에서 필요                |
| `session`   | 유지. follow-up 비용 감소에 필요                   |
| `scope`     | 유지하되 docs/examples에서 강하게 demote           |
| `language`  | wrapper schema에서는 제거하거나 legacy로 demote 권장 |
| `context`   | 가능하면 제거. wrapper별 typed field가 더 좋음       |

`language`는 task text에서 자동 추론한다고 이미 README와 schema가 말합니다. 그러면 wrapper에서는 굳이 계속 노출하지 않아도 됩니다.

`context`는 너무 generic합니다. agent가 모든 애매한 정보를 `context`에 넣기 시작하면 wrapper의 목적형 schema가 흐려집니다.

### 권장 wrapper schema

예를 들어 `find_relevant_code`는 이 정도가 가장 좋습니다.

```json
{
  "query": "string",
  "knownFiles": ["string"],
  "knownSymbols": ["string"],
  "knownText": ["string"],
  "scope": ["string"],
  "repo_root": "string",
  "session": "string"
}
```

그리고 examples에서는 보통 `query`만 보여주는 편이 좋습니다.

```json
{
  "query": "Where is MCP tools/list assembled and what should I read before changing tool metadata?"
}
```

## 3.10 `knownText`가 내부에서 `hints.regex`로 들어갑니다

`buildAnchorHints()`는 `knownText`를 `regex`로 매핑합니다.

```js
const regex = cleanStringArray(knownText);
if (regex.length > 0) hints.regex = regex;
```

근거: `src/mcp/server.mjs:378-386`.

이름은 `knownText`인데 내부는 regex입니다.

### 왜 문제가 되는가

agent가 `knownText: ["C++", "/users/me?tab=profile", "foo.bar"]` 같은 literal text를 넣으면 regex meta character 때문에 의도와 다르게 검색될 수 있습니다.

### 권장 수정

둘 중 하나가 좋습니다.

1. `knownText`는 literal로 escape해서 내부 grep에 넘긴다.
2. advanced field로 `knownRegex`를 따로 만들고, `knownText`는 literal only로 둔다.

agent-facing에서는 regex 사고를 줄이는 것이 목표이므로 1번이 더 낫습니다.

# 4. 불필요한 입력/출력 요소 정리

## 4.1 입력에서 빼거나 숨겨야 할 것

| 요소               | 현재 상태                 | 왜 불필요한가                              | 권장                                         |
| ---------------- | --------------------- | ------------------------------------ | ------------------------------------------ |
| `budget`         | legacy 표시됨            | agent가 turn/latency/coverage를 예측해야 함 | schema 유지, examples 제거, env/debug docs로 이동 |
| `thoroughness`   | legacy 표시됨            | `budget`의 다른 이름                      | `explore` public examples에서 제거             |
| `hints.strategy` | legacy 표시됨            | internal routing을 parent에게 맡김        | wrapper 내부에서만 사용                           |
| `language`       | 아직 wrapper에 노출        | task text에서 자동 추론 가능                 | wrapper schema에서 제거 또는 legacy              |
| `context`        | wrapper마다 존재          | generic field가 목적형 schema를 흐림        | wrapper별 typed input으로 대체                  |
| `scope`          | 계속 노출                 | 잘못 넣으면 탐색을 좁혀 실패함                    | 유지하되 “already known only”로 제한              |
| `hints.regex`    | public에 있음            | agent를 다시 grep 사고로 돌림                | `knownText` literal로 대체                    |
| `maxDepth`       | active docs에서는 제거된 듯함 | 구현 depth와 맞지 않음                      | 계속 비노출 유지                                  |

| 요소                  | 현재 상태     | 왜 불필요한가                          | 권장                                                |
| ------------------- | --------- | -------------------------------- | ------------------------------------------------- |
| `answer`            | top-level | `directAnswer`와 중복               | `_debug.legacy.answer`                            |
| `summary`           | top-level | 대부분 answer와 중복                   | `_debug.legacy.summary` 또는 session only           |
| `candidatePaths`    | top-level | role/action 없음                   | `_debug.legacy.candidatePaths`; agent는 `targets`만 |
| `followups`         | top-level | `nextAction`/`uncertainties`와 중복 | `_debug.legacy.followups`                         |
| `confidenceScore`   | top-level | 과도한 숫자 정밀감                       | `_debug.confidenceScore`                          |
| `confidenceFactors` | top-level | agent가 읽기엔 noisy                 | `_debug.confidenceFactors`                        |
| `critic`            | top-level | `status.warnings`와 중복            | `_debug.critic`                                   |

# 7. 최종 평가

수정 방향은 맞습니다. 특히 다음은 매우 잘 반영됐습니다.

* 목적형 wrapper 추가
* `explore_v2` opt-in 처리
* `budget/thoroughness` legacy demotion
* auto budget 구현
* output schema 추가
* evidence snippets 추가
* `targets/status/nextAction` runtime 연결
* Codex/Claude 주요 instruction 개선
* adoption benchmark 추가
* 테스트 통과

하지만 제가 의도한 “Codex/Claude가 불필요한 판단을 하지 않게 하는 최소 interface”까지 가려면 아직 남은 핵심은 이것입니다.

**v3 필드를 추가하는 것만으로는 부족합니다. legacy/debug 필드를 top-level에서 내려야 합니다.**
지금은 `targets`가 있어도 `candidatePaths`가 보이고, `status`가 있어도 `confidenceScore`가 보이고, `sessionId`가 필요한데 `stats.sessionId`를 읽게 되어 있습니다. 이 상태에서는 agent가 여전히 불필요한 해석을 합니다.

다음 한 번의 개선 목표는 명확합니다.

```txt
MCP structuredContent = compact agent-facing contract
runtime raw result = full legacy/debug contract
```

이 분리를 하면 adoption 관점에서 훨씬 좋아질 것입니다.
