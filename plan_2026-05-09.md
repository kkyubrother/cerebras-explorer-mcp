나의 입력 -> # GPT-5.5-Pro Research Prompt: AI Agent Adoption for cerebras-explorer-mcp

당신은 AI coding agent, MCP tool design, developer workflow adoption을 연구하는 시니어 AI 시스템 설계자입니다.

첨부된 프로젝트 소스코드는 `cerebras-explorer-mcp`입니다.

이 프로젝트는 Codex, Claude Code 같은 상위 AI coding agent가 repo를 직접 수동 탐색하지 않고, MCP 도구를 통해 read-only autonomous explorer
에게 코드 탐색, 심볼 추적, 영향 분석, 근거 수집을 위임하도록 만든 도구입니다. 주요 공개 도구는 `explore_repo`, `explore`, `explore_v2`로 예
상됩니다.

제가 느끼는 문제는 다음과 같습니다.

이 도구는 기능적으로는 잘 만들어졌다고 생각하지만, 실제 AI coding agent들이 이 도구를 적극적으로 사용하지 않고, 여전히 `rg`, `read_file`,
`grep`, 직접 파일 열람 같은 저수준 탐색을 선호하는 것 같습니다.

당신의 연구 목표는 "이 도구가 기술적으로 좋은가?"를 평가하는 것이 아닙니다.

핵심 목표는 다음입니다.

"왜 AI coding agent들이 이 MCP explorer를 자발적으로 잘 선택하지 않는가?"
"어떤 tool name, description, schema, output shape, workflow rule, integration instruction을 바꾸면 agent adoption이 올라가는가?"
"상위 AI가 이 도구 결과를 신뢰하고 중복 탐색을 줄이게 만들려면 무엇이 필요한가?"

첨부된 소스코드를 실제로 읽고, README/문서/도구 스키마/서버 구현/반환 구조/테스트/벤치마크/통합 문서를 근거로 분석하세요. 추측만 하지 말고
가능한 한 파일 경로와 코드 구조를 인용하세요.

## 연구 범위

- 어떤 작업 유형에서는 이 도구가 유리하고, 어떤 작업에서는 직접 탐색이 더 나은지 구분하세요.

### 2. MCP tool naming and description

# A. Executive summary

## 이 도구가 덜 사용되는 가장 큰 이유 5개

### 1. 이름과 description이 “agent가 지금 해야 할 일”보다 “도구가 무엇인지”를 설명한다

`explore_repo`, `explore`, `explore_v2`는 모두 탐색이라는 넓은 동사를 공유합니다. agent 입장에서는 “관련 파일 찾기”, “심볼 추적”, “변경 영
향 분석”, “리뷰 전 context 수집” 같은 즉시 목적형 이름이 더 선택하기 쉽습니다. 현재 `explore_repo` description은 “Use instead of Grep/Glob/
Read”라고 강하게 말하지만, 언제까지나 추상적인 “explore” 계열이라 native `rg`, `read_file`, `grep`보다 선택 트리거가 덜 선명합니다. 근거:
`src/mcp/server.mjs:14-28`, `src/mcp/server.mjs:132-184`.

### 2. 결과가 “믿고 다음 행동으로 넘어가기”보다 “좋은 조사 보고서”에 가깝다

현재 `explore_repo`는 `answer`, `evidence`, `candidatePaths`, `followups`, `confidenceScore`, `critic` 등 좋은 필드를 갖고 있습니다. 하지만
상위 coding agent가 실제로 원하는 상단 필드는 대개 `likelyFilesToEdit`, `nextReadTargets`, `verifiedFacts`, `uncertainties`,
`recommendedNextToolCall`입니다. 현재 text formatter는 `trustSummary`, `Answer`, evidence top 10, followups, stats를 보여주지만, 편집 대상
후보와 다음 read target을 명시적으로 상단에 주지 않습니다. 근거: `src/mcp/server.mjs:298-347`, `DESIGN.md:321-411`.

### 3. native 탐색보다 호출 전 사고 비용이 크다

`explore_repo`는 `task`, `scope`, `budget`, `hints`, `session`, `language`를 받을 수 있고, `explore`/`explore_v2`는 `prompt`,
`thoroughness`를 받습니다. 기능적으로는 유연하지만, agent에게는 “어떤 도구를 고르고, 어떤 budget을 쓰고, hints를 어떻게 넣을지”를 먼저 결정
해야 하는 비용이 됩니다. 반면 native `rg pattern` 또는 `read_file path`는 입력 결정이 매우 작습니다. 근거: `src/explorer/schemas.mjs:1-60`,
`src/mcp/server.mjs:132-184`.

### 4. 신뢰 모델은 잘 시작했지만, 재검증을 줄일 만큼 “작동 가능한 evidence package”가 아니다

이 프로젝트는 evidence grounding과 deterministic critic이라는 좋은 방향을 이미 구현했습니다. 실제 읽은 line range, grep/git 관측 범위만
evidence로 남기고, ungrounded evidence를 drop하며 confidence를 조정합니다. 근거: `DESIGN.md:422-497`, `src/explorer/runtime.mjs:1117-1169`.
하지만 evidence에 snippet, source tool turn, exact observed range, “이건 편집 전 반드시 read해야 함/안 해도 됨” 같은 실행 지침이 부족합니
다. 그래서 상위 agent는 안전하게 다시 파일을 엽니다.

### 5. 현재 benchmark는 explorer 품질을 재지만 adoption을 재지 않는다

`benchmarks/core.json`에는 6개 케이스가 있고, evaluator는 keyword coverage, evidence count, candidate path count, grounded evidence 등을 점
수화합니다. 근거: `benchmarks/core.json:7,64,123,184,238,289`, `src/benchmark/evaluator.mjs:12-152`. 그러나 “agent가 먼저 explorer를 골랐는
가”, “native read/grep이 줄었는가”, “explorer evidence를 다시 읽는 중복률이 줄었는가” 같은 adoption metric은 없습니다. 따라서 기능 개선이
실제 tool-selection 행동을 바꾸는지 알기 어렵습니다.

# B. Source-grounded diagnosis

## 1. 현재 설계의 강점

이 프로젝트는 “read-only autonomous explorer”라는 정체성이 분명합니다. README는 상위 AI가 반복적인 파일 탐색에 context를 쓰지 않고
`explore_repo`, `explore`, `explore_v2` 호출로 검증된 코드 근거와 요약을 받는 것을 핵심 목적으로 설명합니다. 근거: `README.md:5-9`.

내부 구조도 adoption에 필요한 기반을 갖고 있습니다. 상위 모델은 MCP tool을 호출하고, 서버 내부 Cerebras model이 `repo_list_dir`,
`repo_find_files`, `repo_grep`, `repo_symbols`, `repo_references`, `repo_read_file`, git 도구를 사용해 autonomous loop를 수행합니다. 근거:
`README.md:98-119`.

특히 좋은 부분은 다음입니다.

| 강점                             |                                               근거 | adoption 관점의 의미           |
  | ------------------------------ | -----------------------------------------------: | ------------------------- |
| read-only, bash/write 없음       |                              `README.md:123-125` | 상위 agent가 안전하게 위임할 수 있음   |
| evidence grounding             |         `README.md:132-133`, `DESIGN.md:422-439` | hallucinated citation을 줄임 |
| deterministic critic           |                              `DESIGN.md:443-497` | “어디까지 믿어도 되는지”를 구조화할 수 있음 |
| session continuation           |         `README.md:128-129`, `DESIGN.md:224-230` | 후속 탐색에서 반복 비용 감소          |
| specialized shortcuts 존재       | `README.md:276-290`, `src/mcp/server.mjs:33-128` | wrapper tool 전략으로 확장하기 쉬움 |
| Codex/Claude integration 파일 존재 |  `integrations/codex/*`, `integrations/claude/*` | adoption layer를 이미 고려함    |

문제는 “기술적 가능성”이 아니라, 상위 agent가 tool selection 순간에 이 강점을 충분히 보지 못한다는 점입니다.

  agents/cerebras-explorer.md:16-26`                              | “prefer”는 native tool habit을 이기기 어렵다

## 3. 왜 Codex/Claude류 agent는 native 탐색을 선호하는가

### Native `rg/read_file`의 장점

Native 탐색은 작고, 즉시적이고, 결정론적입니다. `rg requireAuth`는 pattern 하나만 정하면 되고, `read_file src/x.ts`는 path 하나만 정하면 됩
니다. 반면 `explore_repo`는 task를 문장으로 만들어야 하고, scope/budget/hints를 고를 수 있어 agent의 planning cost가 커집니다.

또한 coding agent는 최종적으로 코드를 수정해야 합니다. Explorer가 아무리 좋은 근거를 주어도 edit 전에 target file을 직접 읽어야 하는 상황이
많습니다. 따라서 “explorer 호출 + 다시 read”가 “그냥 처음부터 read”보다 싸다는 확신이 없으면 native 탐색으로 갑니다.

### 현재 explorer가 유리한 작업

Explorer 우선이 맞는 작업은 다음입니다.

| 작업 유형                                  | 이유                                                         |
  | -------------------------------------- | ---------------------------------------------------------- |
| 정확한 파일을 모르는 기능/버그 조사                   | 내부 autonomous loop가 후보 파일 압축 가능                            |
| 3개 이상 파일을 읽어야 할 것 같은 task              | parent context 낭비를 줄임                                      |
| architecture / subsystem overview      | `explore`/`explore_v2` report가 유리                          |
| symbol definition + usage + caller 추적  | `explain_symbol`, `trace_dependency`, `repo_references` 기반 |
| route/middleware/config origin tracing | grep + symbols + read loop 조합 필요                           |
| impact mapping before edits            | candidate path와 cross-file evidence가 필요                    |
| git-guided change summary              | git tools가 내부에 있음                                          |
| repeated pattern / similar code search | regex보다 reasoning 기반 scan이 유리                              |

### 직접 탐색이 더 나은 작업

| 작업 유형                            | 이유                                                                            |
  | -------------------------------- | ----------------------------------------------------------------------------- |
| 이미 정확한 file path와 line range가 있음 | `read_file` 1회가 더 빠름                                                          |
| 단일 파일 수정                         | edit 전 직접 read가 어차피 필요                                                        |
| compiler/test error가 특정 파일/라인을 줌 | high-level 탐색보다 target read가 효율적                                              |
| 사용자가 raw local verification을 요구  | explorer는 외부 model call이므로 부적절할 수 있음                                          |
| API key/latency/failure 가능성이 문제  | README상 Cerebras Inference API 호출 기반임. 근거: `README.md:3-5`, `README.md:31-40` |
| 아주 작은 repo 또는 1-2 파일 task        | delegation overhead가 더 큼                                                      |

# C. Rewritten MCP tool descriptions

아래 description은 `src/mcp/server.mjs`에 바로 넣을 수 있는 형태입니다. 기존 tool name은 유지해 backward compatibility를 보존합니다.

## `explore_repo`

  ```js
  description:
    'Use FIRST for read-only code discovery when the exact files are unknown, the task may span 3+ files, or you need cross-file
  understanding: architecture, symbol usage, dependency/call tracing, bug root cause, change impact, config origin, or evidence collection.
  Do NOT use for a single known file/range or when immediate editing is cheaper. Returns structured JSON with answer, grounded file:line
  evidence, candidate paths, confidence/critic warnings, stats, and follow-up suggestions. After this tool, avoid broad grep/read; only read
  cited targets needed for verification or edits. Pass stats.sessionId as "session" for follow-up calls.'
  ```

## `explore`

  ```js
  description:
    'Use for a user-facing Markdown investigation report with inline file:line citations. Best for architecture walkthroughs, onboarding
  explanations, code review context, or “how does X work?” answers. Do NOT use when the parent agent needs structured edit planning or
  programmatic next steps; use explore_repo or a purpose-specific wrapper instead. Use quick for narrow reports, normal for scoped reports,
  deep for broad subsystem reports.'
  ```

## `explore_v2`

  ```js
  description:
    'Use only for wide or deep Markdown reports that may exceed normal context/output limits. It performs compaction, tool-output budgeting,
  and output recovery. Best for large architecture deep-dives, end-to-end root-cause reports, or broad subsystem maps. Do NOT use just
  because the file path is unknown; prefer explore_repo or a purpose-specific wrapper when structured evidence, likely edit files, or next
  read targets are needed.'
  ```

# D. Proposed wrapper tools

권장 방향은 **기존 `explore_repo`, `explore`, `explore_v2`를 유지하고, 목적형 wrapper를 추가**하는 것입니다. 이유는 세 가지입니다.

첫째, backward compatibility를 유지합니다. 둘째, 현재 내부 구현이 이미 `explore_repo`에 task/hints를 주입하는 wrapper 방식을 사용합니다. 근
거: `src/mcp/server.mjs:213-268`. 셋째, agent tool selection은 name과 description의 영향을 강하게 받으므로, “무엇을 하라”가 이름에 들어간
wrapper가 adoption에 유리합니다.

아래 wrapper들은 모두 read-only입니다.

  ---

## 1. `find_relevant_code`

### Purpose

사용자 task와 관련된 파일, entry point, symbol, candidate edit location을 빠르게 찾습니다.

### When to use

정확한 파일을 모르고, 여러 파일을 읽어야 할 것 같을 때. 구현 전에 “어디를 봐야 하지?”라는 단계에서 사용합니다.

### When not to use

이미 수정할 파일이 정확히 알려져 있고 1-2회 read로 충분할 때.

### Input schema

  ```json
  {
    "type": "object",
    "additionalProperties": false,
    "required": ["query"],
    "properties": {
      "query": { "type": "string" },
      "repo_root": { "type": "string" },
      "scope": { "type": "array", "items": { "type": "string" } },
      "knownFiles": { "type": "array", "items": { "type": "string" } },
      "knownSymbols": { "type": "array", "items": { "type": "string" } },
      "knownRegex": { "type": "array", "items": { "type": "string" } },
      "maxTargets": { "type": "integer", "default": 8 },
      "thoroughness": { "type": "string", "enum": ["quick", "normal", "deep"], "default": "normal" },
      "session": { "type": "string" },
      "language": { "type": "string" }
    }
  }
  ```

### Output schema

  ```json
  {
    "directAnswer": "string",
    "relevantFiles": [
      {
        "path": "string",
        "role": "entrypoint|implementation|caller|test|config|doc|unknown",
        "reason": "string",
        "evidenceRefs": ["E1"],
        "confidence": "low|medium|high"
      }
    ],
    "likelyFilesToEdit": [],
    "nextReadTargets": [],
    "evidence": [],
    "verifiedFacts": [],
    "uncertainties": [],
    "recommendedNextToolCall": null,
    "stats": {}
  }
  ```

### Internal implementation strategy

Internally call `explore_repo` with:

  ```js
  {
    task: `Find the code most relevant to: ${query}. Return likely files to inspect/edit and evidence.`,
    budget: thoroughness,
    hints: {
      files: knownFiles,
      symbols: knownSymbols,
      regex: knownRegex,
      strategy: knownSymbols?.length ? 'symbol-first' : 'breadth-first'
    }
  }
  ```

### Migration compatibility

Additive only. Existing `explore_repo` users unaffected.

## 2. `trace_symbol`

### Purpose

심볼의 정의, 사용처, caller/callee, 관련 파일을 추적합니다. 현재 `explain_symbol`보다 agent가 더 자연스럽게 선택할 이름입니다.

### When to use

함수/class/type/constant가 어디서 정의되고 어디서 쓰이는지 알아야 할 때.

### When not to use

단순 문자열 검색이 충분하거나 symbol name이 너무 모호해서 query를 먼저 좁혀야 할 때.

### Input schema

  ```json
  {
    "type": "object",
    "additionalProperties": false,
    "required": ["symbol"],
    "properties": {
      "symbol": { "type": "string" },
      "kind": { "type": "string", "enum": ["function", "class", "type", "variable", "constant", "unknown"] },
      "startingFile": { "type": "string" },
      "direction": { "type": "string", "enum": ["definition", "usages", "callers", "callees", "both"], "default": "both" },
      "repo_root": { "type": "string" },
      "scope": { "type": "array", "items": { "type": "string" } },
      "session": { "type": "string" },
      "language": { "type": "string" }
    }
  }
  ```

`maxDepth`는 조심해야 합니다. 현재 design은 `repo_symbol_context`의 depth interface가 있어도 실질적으로 `effectiveDepth = 1`로 고정된다고
명시합니다. 근거: `DESIGN.md:289-295`. 따라서 지금 당장 wrapper schema에 `maxDepth`를 넣는다면 실제 구현도 반복 reference chase를 해야 합니
다. 아니면 노출하지 않는 편이 안전합니다.

### Output schema

  ```json
  {
    "directAnswer": "string",
    "definitions": [
      { "symbol": "string", "path": "string", "range": [1, 10], "evidenceRefs": ["E1"] }
    ],
    "usages": [
      { "path": "string", "range": [20, 25], "usageKind": "import|definition|call|reference", "evidenceRefs": ["E2"] }
    ],
    "callPath": [],
    "nextReadTargets": [],
    "evidence": [],
    "uncertainties": [],
    "confidenceScore": 0.0
  }
  ```

### Internal implementation strategy

Alias to existing `explain_symbol` or call `explore_repo` with `hints.strategy = "symbol-first"` and `hints.symbols = [symbol]`.

### Migration compatibility

Keep `explain_symbol`; expose `trace_symbol` as clearer alias. Later deprecate `explain_symbol` only in docs, not in server.

# E. Output schema redesign

## 유지할 것

| Field                                                | 판단                                                         |
  | ---------------------------------------------------- | ---------------------------------------------------------- |
| `answer` / `summary`                                 | 유지. 다만 `directAnswer`를 alias로 상단에 추가                       |
| `confidence`, `confidenceScore`, `confidenceFactors` | 유지. 이미 구현 의도가 좋음. 근거: `DESIGN.md:327-342`                  |
| `evidence`                                           | 유지하되 강화                                                    |
| `candidatePaths`                                     | 유지. 단, 역할/우선순위가 없는 raw path list로는 부족                      |
| `followups`                                          | 유지. 하지만 `recommendedNextToolCall`로 one-best action을 상단에 추가 |
| `critic`                                             | 유지. deterministic critic 방향은 매우 좋음                         |
| `stats.sessionId`                                    | 유지. follow-up 비용 감소에 중요                                    |
| `recentActivity`                                     | git-guided task에서 유지                                       |
| `codeMap`, `diagram`                                 | 유지하되 기본 상단 필드에서는 내림                                        |

## 줄일 것

| Field/형태                           | 이유                                               |
  | ---------------------------------- | ------------------------------------------------ |
| text response의 긴 narrative         | coding agent는 다음 행동 후보를 먼저 원함                    |
| `candidatePaths`만 있는 raw file list | 역할과 edit likelihood가 없으면 다시 grep/read하게 됨        |
| generic `followups`                | 여러 follow-up보다 “다음에 이걸 하라”가 adoption에 유리         |
| `explore_v2` 내부 구현 설명 중심           | tool selection 순간에는 implementation보다 trigger가 중요 |

## 새로 추가할 것

요청하신 필드는 대부분 필요합니다.

| Field                     | 필요성   | 권장                                                 |
  | ------------------------- | ----- | -------------------------------------------------- |
| `directAnswer`            | 매우 높음 | `answer` alias로 상단 배치                              |
| `evidence`                | 필수    | `id`, `snippet`, `sourceTool`, `observedAtTurn` 추가 |
| `nextReadTargets`         | 매우 높음 | 중복 탐색을 줄이는 핵심                                      |
| `likelyFilesToEdit`       | 매우 높음 | coding workflow adoption 핵심                        |
| `verifiedFacts`           | 높음    | “이건 확인됨”과 “추정” 분리                                  |
| `uncertainties`           | 매우 높음 | 재검증이 필요한 범위만 좁힘                                    |
| `recommendedNextToolCall` | 높음    | follow-up 선택 비용 감소                                 |
| `confidenceScore`         | 유지    | 이미 구현됨                                             |
| `confidenceFactors`       | 유지    | 상단에는 compact, 상세는 하단                               |
| `toolCallTrace`           | 추가 권장 | full transcript가 아니라 compact trace                 |
| `resultContractVersion`   | 추가 권장 | client parsing 안정성                                 |

## agent가 가장 먼저 읽어야 하는 상단 5개 필드

1. `directAnswer`
2. `confidenceSummary`
3. `likelyFilesToEdit`
4. `nextReadTargets`
5. `evidence`

`confidenceSummary`는 `confidenceScore`, `confidenceLevel`, `critic.status`, 핵심 `confidenceFactors`를 합친 compact object가 좋습니다.

MCP 측면에서는 이 schema를 tool definition의 `outputSchema`로 노출하는 것이 중요합니다. MCP spec은 structuredContent와 outputSchema가
client/LLM의 parsing과 validation을 돕는다고 설명합니다. ([Model Context Protocol][1])

## evidence trust model

상위 agent가 중복 탐색을 줄이려면 다음 규칙이 필요합니다.

| 조건                                                            | parent 행동                                                   |
  | ------------------------------------------------------------- | ----------------------------------------------------------- |
| `critic.status = pass`, `groundingStatus = exact`, snippet 있음 | broad grep/read 금지. `nextReadTargets`만 읽음                   |
| `groundingStatus = partial`                                   | 해당 evidence range만 target read                              |
| `critic.status = caution`                                     | `uncertainties`와 `recommendedNextToolCall`만 follow-up       |
| `critic.status = fail`                                        | explorer 결과를 orientation으로만 사용하고 native verification 또는 재탐
  색 |
| edit 직전                                                       | `likelyFilesToEdit`의 실제 수정 range는 직접 read 후 edit            |

목표는 “맹신”이 아니라 “검증 범위를 줄이는 것”입니다.

# F. Agent instruction pack

Codex는 `AGENTS.md`를 작업 전에 읽고, global/repo/subdirectory instruction을 계층적으로 합칩니다. ([OpenAI 개발자][2]) Codex skill은 처음에
는 skill의 name/description/path만 context에 있고, task가 description과 맞을 때 full `SKILL.md`를 읽습니다. 그래서 description은 짧고
trigger words를 앞에 둬야 합니다. ([OpenAI 개발자][3]) Claude Code는 subagent description을 보고 위임 시점을 결정하며, 자체 Explore
subagent도 read-only codebase search/exploration 용도로 존재합니다. ([Claude][4]) 따라서 이 프로젝트의 instruction pack은 “문서 예쁘게 쓰
기”보다 **첫 번째 tool 선택을 바꾸는 짧은 규칙**이어야 합니다.

  ---

## 1. Codex `AGENTS.md`에 넣을 문구

  ```md
  ## Cerebras Explorer first-pass rule

  Use the `cerebras-explorer` MCP tools before native wide search when the exact file is unknown or the task may require reading 3+ files.

  Use explorer first for:
  - unfamiliar subsystem discovery
  - architecture or code path explanation
  - symbol definition/usage tracing
  - route, middleware, dependency, or config-origin tracing
  - change-impact mapping before edits
  - PR/review context collection
  - evidence collection with file:line citations

  Preferred flow:
  1. Call the narrowest purpose tool available: `find_relevant_code`, `trace_symbol`, `map_change_impact`, `explain_code_path`,
  `collect_evidence`, `review_change_context`, or existing `explore_repo`.
  2. Treat explorer output as the orientation map.
  3. Do not run broad `rg`/grep/glob after a passing explorer result.
  4. Read only `nextReadTargets`, cited evidence ranges, or `likelyFilesToEdit` before editing.
  5. Reuse `stats.sessionId` as `session` for follow-up explorer calls.

  Do not use explorer by default when:
  - the exact file and line range are already known
  - one or two direct reads are clearly cheaper
  - the task is mainly to edit a known file
  - the user asks for raw local verification first
  - the explorer result is thin, contradictory, or marked `critic.status: fail`
  ```

  ---

## 2. Codex skill `SKILL.md`에 넣을 문구

  ```md
  ---
  name: cerebras-explore
  description: Use first for broad read-only code discovery when exact files are unknown, 3+ files may be needed, or cross-file architecture/
  symbol/impact evidence is required. Do not use for one known file/range.
  ---

  Use this skill to avoid spending many native grep/read steps on repository discovery.

  Trigger this skill for:
  - find relevant code before implementation
  - trace symbol definitions/usages/callers
  - map change impact before edits
  - explain route, middleware, dependency, or config paths
  - collect cited evidence
  - review PR/change context

  Workflow:
  1. Choose the most specific Cerebras Explorer tool.
  2. Keep the query close to the user's wording.
  3. Add scope, known files, known symbols, or regex only if already known.
  4. Prefer `normal` for scoped discovery and `deep` for unfamiliar subsystems.
  5. After a result with `critic.status: pass`, do not perform broad native search.
  6. Read only `nextReadTargets`, evidence ranges, or likely edit files before making changes.
  7. Use native wide search only when the result is insufficient or the task explicitly requires local verification.
  ```

현재 repo의 Codex skill example은 `trace_dependency({ ..., maxDepth: 3 })`를 보여주지만 실제 schema는 `maxDepth`를 허용하지 않습니다. 근거:
`integrations/codex/.agents/skills/cerebras-explore/SKILL.md:39-40`, `src/mcp/server.mjs:63-82`. 이 부분은 즉시 고쳐야 합니다.

## 3. Claude Code custom agent 또는 skill 문구

Claude file-based subagent는 `description`, `tools`, `disallowedTools`, `permissionMode`, `mcpServers` 같은 frontmatter field를 지원합니다.
([Claude][4]) 현재 repo의 Claude agent는 이미 `mcpServers: - cerebras-explorer`와 write/edit 금지를 넣고 있습니다. 근거: `integrations/
  claude/.claude/agents/cerebras-explorer.md:1-8`. 여기에 trigger를 더 강하게 넣는 것이 좋습니다.

  ```md
  ---
  name: cerebras-explorer
  description: Use first for read-only repository discovery when exact files are unknown, 3+ files may be needed, or cross-file architecture/
  symbol/impact evidence is required. Returns only grounded context to the parent.
  model: inherit
  permissionMode: plan
  mcpServers:
    - cerebras-explorer
  disallowedTools: Write, Edit, MultiEdit, NotebookEdit
  ---

  You are a read-only repository exploration coordinator.

  Rules:
  1. Call Cerebras Explorer before native wide Grep/Glob/Read for broad discovery.
  2. Prefer purpose tools in this order:
     - find_relevant_code
     - trace_symbol
     - map_change_impact
     - explain_code_path
     - collect_evidence
     - review_change_context
     - explore_repo
     - explore / explore_v2 only for user-facing reports
  3. Return:
     - direct answer
     - likely files to edit
     - next read targets
     - key evidence with file:line
     - uncertainties
  4. Do not modify files.
  5. Do not ask the parent to run broad search unless the explorer result is thin, conflicting, or failed.
  ```

Claude skill wrapper:

  ```md
  ---
  name: cerebras-explore
  description: Delegate broad read-only code discovery to the cerebras-explorer agent before native search when the task spans unknown files,
  3+ files, architecture, symbols, dependencies, impact, or evidence collection.
  context: fork
  agent: cerebras-explorer
  ---

  Delegate discovery to the cerebras-explorer agent.
  The parent should only verify cited ranges or edit targets after the explorer returns.
  ```

## 4. 일반 MCP client 문서에 넣을 문구

  ```md
  ### When to call Cerebras Explorer

  Call Cerebras Explorer before manual file search when:
  - you do not know the exact file path
  - the question may require 3 or more files
  - the task requires cross-file understanding
  - you need line-cited evidence before editing or reviewing code

  Do not call it for:
  - one known file or line range
  - immediate edits to an already-open file
  - exact compiler/test errors pointing to a single location

  After a successful result:
  - trust `verifiedFacts` with exact evidence
  - inspect only `nextReadTargets` before editing
  - avoid repeating broad grep/read unless `critic.status` is caution/fail
  ```

## 5. 사용 금지/비권장 상황 문구

  ```md
  Do not use Cerebras Explorer as the default for small local tasks.
  Prefer direct native reads when:
  - the user named the exact file
  - the file is already open or recently read
  - the task is a simple one-file edit
  - a test/compiler error gives an exact file and line
  - the user asks for local-only verification
  - external model calls are unavailable or inappropriate
  ```

## 결론

`cerebras-explorer-mcp`는 내부적으로 이미 “좋은 read-only autonomous explorer”의 핵심 요소를 갖고 있습니다. README와 DESIGN이 말하는 목표도
